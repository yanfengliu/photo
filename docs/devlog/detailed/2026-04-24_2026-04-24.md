# Detailed Devlog

## [2026-04-24 10:07 PDT] - Move contrast pivot to perceptual space and reposition tone zones
**Action:** Addressed user feedback that contrast, highlights, shadows, whites, and blacks felt unnatural. Two root causes:
1. `apply_contrast` (CPU `src/edit.rs` and the contrast block in `assets/shaders/image.wgsl`) computed the sigmoid in linear RGB with pivot at `0.5` — which sits at L\* ≈ 76, not middle gray. At `+0.5` contrast a linear 0.218 pixel (L\* 50, middle gray) dropped to 0.157 (a 28% darkening), producing the "image gets duller, not more punchy" symptom.
2. `apply_tone_zones` placed the four zones at L\_p edges that didn't match photographer semantics: Shadows peaked at L\_p 0.22 (near-black), Highlights at 0.55 (bright midtones), Whites started at 0.60, Blacks fell off by 0.30 — so Shadows/Blacks overlapped heavily and Highlights/Whites leaked into midtones.

Rewrote both on CPU and GPU:
- **Contrast** now operates in gamma-2.2 perceptual space with the same k-sigmoid shape. The pivot at `L_p = 0.5` corresponds to L\* 50 (`L_lin ≈ 0.218`). HDR values still normalize through the same per-pixel `peak` step. Identity at middle gray is now exact.
- **Tone zones** now use:
  - Blacks: `1 - smoothstep(0.0, 0.20, L_p)` (peaks near-black, fades by L\_p 0.20)
  - Shadows: `smoothstep(0.10, 0.30) * (1 - smoothstep(0.40, 0.55))` (bell peaks L\_p ~0.32)
  - Highlights: `smoothstep(0.50, 0.65) * (1 - smoothstep(0.80, 0.95))` (bell peaks L\_p ~0.72)
  - Whites: `smoothstep(0.85, 1.0, L_p)` (rises in near-white only)

TDD order:
1. Added `apply_contrast_preserves_middle_gray_at_full_strength` — pre-fix reported Δ 28%; post-fix exact identity at pivot.
2. Added `tone_zones_highlights_do_not_leak_into_lower_midtones` (asserts L\_p 0.40 receives no highlights effect) and `tone_zones_whites_do_not_leak_into_midtones` (asserts L\_p 0.65 receives no whites effect). Both failed pre-fix because highlights previously started at L\_p 0.35 and whites at 0.60.
3. Updated `tone_zones_highlights_affects_bright_not_dark` to use `[0.48, 0.48, 0.48]` (L\_p 0.68, inside new highlights bell) instead of `[0.9, 0.9, 0.9]` (L\_p 0.95, now Whites territory).
4. Updated `tone_zones_shadows_affects_dark_not_bright` to use `[0.08, 0.08, 0.08]` (L\_p 0.33, inside new shadows bell) instead of `[0.02, 0.02, 0.02]` (L\_p 0.17, borderline blacks).
5. Updated `tone_zones_blacks_darkens_dark_pixels` to use `[0.005, 0.005, 0.005]` (L\_p 0.09, inside new blacks zone) instead of `[0.02, 0.02, 0.02]`. The `[0.02]` pixel now sits at the edge of the Blacks fade where weight drops to 6.5% — test failed with 8.6% darkening against the >10% threshold until the test pixel moved inside the zone peak.

## [2026-04-24 10:51 PDT] - Port contrast and tone-zone math to match darktable
**Action:** User followed up on the previous tuning change noting "is this how Adobe Lightroom does it?" — an honest re-review showed the previous commit's wording ("Lightroom-style") overstated the relationship (Lightroom's math is proprietary). Switched the reference to darktable, which has an open, well-documented implementation. Fetched `src/iop/basicadj.c` and `src/iop/toneequal.c` from `darktable-org/darktable` to get the actual formulas, then rewrote both operators on CPU (`src/edit.rs`) and GPU (`assets/shaders/image.wgsl`) to match:

- **Contrast** is now darktable's basicadj per-channel power law in scene-referred linear RGB:
  `output = (input / 0.1842) ^ (1 + amount) * 0.1842`.
  The pivot `0.1842` is darktable's default middle gray — the linear Y of CIELab `L* 50`. At amount = 0 the expression is exact identity; above the pivot the curve amplifies and below it compresses. HDR values beyond 1.0 simply amplify further under +contrast (no peak-normalization clamp) because that is how a scene-referred pipeline is expected to behave.
- **Tone zones** are now darktable's tone-equalizer Gaussian bands in log₂ luminance. Darktable ships 9 bands spanning -8..0 EV; our 4-slider UI collapses those into Zone System positions: Blacks at -7 EV, Shadows at -4 EV, Highlights at -1 EV, Whites at 0 EV. Each band contributes `exp(-(ev - center)² / (2σ²))` with σ = √2 (darktable's default), multiplied by the slider's UI value × 2 (so UI ±1 maps to ±2 EV per band). The sum is clamped to ±2 EV, matching darktable's [0.25, 4.0] linear correction limit. The wide Gaussian bands are intentional — adjacent zones overlap smoothly so the tonal curve stays continuous across the full EV range. That's why `tone_zones_highlights_do_not_leak_into_lower_midtones` and `tone_zones_whites_do_not_leak_into_midtones` (the previous commit's "no leakage" tests) were removed and replaced with peak-position tests (`tone_zones_highlights_peak_at_bright_midtones_not_shadows`, `tone_zones_whites_peak_near_white_not_midtones`) — the contract now is about where the peak response sits, not about zero leakage between bands.

Updated the other existing tests for the new contract: the contrast pivot test now uses `0.1842`, the positive/negative-contrast tests now use `[0.1]` as the below-pivot sample (0.2 sat above the 0.1842 pivot), the HDR contrast test now asserts scene-referred amplification instead of near-identity preservation, and the shadows-minimal-on-bright test now tolerates 5% rather than 2% to acknowledge the Gaussian tail.

User followed with "not just contrast but all params" during implementation. Audit of the remaining five sliders:

- **Exposure** (`apply_exposure`) is already `px * 2^ev` which is byte-for-byte what darktable's exposure module does. No change needed.
- **Temperature/Tint** (`temperature_tint_matrix`) uses a Bradford CAT matrix built from CIE D-illuminant chromaticity. Darktable's `temperature` module uses the same CAT family (Bradford or CAT02). No change needed.
- **Vibrance** (`apply_vibrance`) was the one meaningful scope decision. Darktable's `vibrance.c` works in Lab space with `ss = 1 + amount * sqrt(a²+b²)/256`, i.e. it *amplifies* already-saturated pixels. That is the opposite of Lightroom/common-usage "Vibrance" which *protects* saturated pixels and boosts muted ones, which is what our current power-law attenuation implementation already does. Porting darktable's literal formula would invert user expectations for a slider labelled "Vibrance", so we intentionally keep the existing Lightroom-style implementation and document the divergence here.
- **Saturation** (`apply_saturation`) uses a linear blend from luminance, `px = lum + (px - lum) * (1 + amount)`. Darktable's modern saturation lives in `colorbalancergb.c` and operates in Oklab/YCbCr-like chroma space — a substantially larger refactor for modest perceptual improvement. Keep the current implementation; note this is a future improvement opportunity if the user reports that saturation feels unnatural.
- **Clarity** (`apply_all` clarity branch) uses `px += amount * (px - blur) * midtone` with a Gaussian blur and midtone-smoothstep mask. Darktable's `local_contrast.c` uses a bilateral/guided filter for edge-aware blur, which is significantly more expensive and requires a different GPU pipeline. The current simplified form is the common "photographer clarity" approach and keeps the real-time preview affordable.
- **Dehaze** (`apply_all` dehaze branch) uses a simplified dark-channel prior with the local blur as the per-pixel atmospheric estimate. Darktable's `hazeremoval.c` implements the full He et al. dark-channel prior with guided filter refinement. The full method requires global atmospheric-light estimation (a whole-image reduction) and guided filtering of a transmission map — both unsuitable for a per-frame GPU preview. The current simplified form stays.

Summary: the two operators the user named as feeling unnatural (contrast plus the four tone-zone sliders) are now faithful darktable ports; the rest either already match darktable's math or keep a simpler implementation for real-time constraints or to match photographer expectations under the existing slider labels.
